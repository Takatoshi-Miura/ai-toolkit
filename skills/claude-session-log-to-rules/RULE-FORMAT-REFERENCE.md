# ルールファイル書式リファレンス

ルール化候補をファイルに反映する際の書式規約。Phase 2（プロジェクト名/グローバル判定）とPhase 4（編集）で参照する。

## 編集対象ファイル

`~/.claude/` 配下を直接編集する。

- グローバルルール: `~/.claude/CLAUDE.md`
- プロジェクト別ルール: `~/.claude/rules/<project-kebab-case>.md`

## プロジェクト名 vs グローバルの判定基準

候補の`cwd`/`project_dir_name`/`context_excerpt`を見て、以下の優先順位で判定する。

1. **特定プロジェクトに閉じた具体的指示**（特定ライブラリ・API名・特定画面/機能への言及、特定言語・フレームワーク固有の作法）→ プロジェクト別ルール
   - 例: 「MenuはSettingにあるべき」（SportsNote-iOSのアーキテクチャ規約）
2. **言語・プロジェクトを問わない汎用的な開発プラクティス**（コミットメッセージ規約、コミュニケーション様式、共通ツールの使い方）→ グローバルルール（`CLAUDE.md`）
   - 例: 「日本語で回答すること」「読み取り系コマンドは確認なしで許可」
3. `cwd`がホームディレクトリ直下や`Downloads`など特定プロジェクトに紐づかない場所で、内容も雑談・個人的相談（住宅ローン相談等）の場合 → ルール化対象外（候補から除外）

判定に迷う場合はユーザーに確認する。

## プロジェクト別ルールファイルの書式

既存ファイル（`rules/android-development.md`, `rules/swift-testing.md`）を参考例とする。

```markdown
---
paths:
  - "**/*.kt"
  - "**/res/**/*.xml"
---

# {プロジェクト名}開発ルール

{1行概要}

## {テーマ別セクション}

- 箇条書きで具体的・検証可能な指示を記載
- 抽象的な精神論ではなく、具体的なAPI名・ファイル名・手順を書く
```

- frontmatterの`paths`は対象プロジェクトの主要言語・フレームワークから推定する
  - Kotlin/Android → `**/*.kt`, `**/res/**/*.xml`, `**/build.gradle*`
  - Swift/iOS → `**/*.swift`
  - Python → `**/*.py`
  - 言語非依存の規約（コミット規則・PRレビュー観点等）はそのプロジェクトの典型的な対象パスを広めに指定する（例: `**/*`）
- ファイル名は`rules/<project-kebab-case>.md`（例: `SportsNote-iOS` → `sportsnote-ios.md`）。既存ファイルは`<topic>.md`（`swift-testing.md`等）の命名だが、本スキルが新規作成するファイルは「プロジェクト名」を採用する
- 本文は50〜60行程度に収める。H1タイトル→H2セクション（テーマ別）→箇条書き
- 既存セクションに近いテーマの追記は新規H2を作らず箇条書きを追記する。新テーマなら新規H2を作る

## グローバルルール（CLAUDE.md）の書式

frontmatterは不要。H2セクションで分類し、汎用的な指示のみ記載する。

```markdown
# グローバル設定

全プロジェクトに適用されるルールをここに記述します。

## {テーマ}

- 箇条書きで具体的に記載
```

## 追記基準のチェック

- 1セッションだけで言われた一時的な指示（その場限りのファイル名・パス指定）はルール化しない
- 複数セッション・複数プロジェクトで繰り返し言われている指摘を優先する
- スキル定義文やコマンド実行結果がユーザー発言として混入しているもの（`# XXX Skill`のような見出しで始まる長文、`Base directory for this skill:`で始まる文章）はルール化候補から除外する。これはユーザーの発言ではなく、スキル実行時にシステムが注入した文章である
