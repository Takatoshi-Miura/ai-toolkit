# 重複・矛盾チェックリファレンス

Phase 3で、ルール化候補と既存ルールファイル（`~/.claude/CLAUDE.md`、関連する`~/.claude/rules/*.md`）を比較し、以下の4分類に振り分けるためのチェックリスト。

## 分類

### 1. 新規追加
対象ファイルに同趣旨の記載が存在しない。新規セクションまたは新規ファイルとして追加する。

### 2. 既存セクションへの追記
対象ファイル内に近いテーマのH2セクションが既にある。そのセクションに箇条書きを追記する（新規H2は作らない）。

### 3. 矛盾（要確認）
既存ルールと候補の内容が直接矛盾する（例: 既存ルールが「Aを使う」、候補が「AではなくBを使う」）。
- 候補のタイムスタンプと既存ルールの最終更新（git log等で確認できる場合）を比較し、どちらが新しい方針かをユーザーに提示する
- 自動で上書きせず、必ずユーザーに「既存ルールを更新するか」を確認する

### 4. 完全重複（スキップ）
文言が違っても意味が実質的に同じ内容が既に書かれている。ルール化せず候補を捨てる。

## 判定の進め方

1. 候補をテーマ別に集約した後、各テーマについて対象スコープ（プロジェクト別 or グローバル）に該当する既存ファイルを`Read`する
2. ファイル内の既存箇条書きと候補を意味レベルで比較する（文言一致ではなく趣旨の一致を見る）
3. 上記4分類のいずれかに割り振る

## 「一時的な指示」と「再発する規約」の区別

ルール化すべきは後者のみ。判断基準:

| 一時的な指示（ルール化しない） | 再発する規約（ルール化する） |
|---|---|
| 特定のファイル・パスを指定した単発の作業依頼（例: 「このファイルを直して」） | コーディングスタイル・命名規則（例: 「テストはSwift Testingを使う」） |
| その場の確認・相づち（例: 「OK」単体） | レビュー観点・品質基準（例: 「@MainActorを付与する条件」） |
| 個人的な相談・雑談（住宅ローン相談、旅行先提案等、開発と無関係） | コミュニケーション様式（例: 「日本語で回答する」） |
| セッション内で完結した一度限りの方針転換 | 複数セッション・複数プロジェクトで繰り返し言われている指摘 |

## 優先度の重み付け

- 同趣旨の指摘が複数セッション・複数プロジェクトで見つかった場合は優先度を上げ、ルール化を強く推奨する
- 1セッション内の1回だけの発言は、内容が明確に再発しそうな規約（具体的なAPI名・フレームワーク作法の指定など）でない限り、ユーザーに確認を取ってから採用する
