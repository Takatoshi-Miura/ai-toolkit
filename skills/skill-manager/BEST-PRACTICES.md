## 基本原則

### 簡潔さが鍵
コンテキストウィンドウは共有リソース。各情報に問う：
- 「Claudeは本当にこの説明が必要か？」
- 「Claudeがこれを知っていると仮定できるか？」
- 「この段落はトークンコストに見合うか？」

**デフォルトの前提: Claudeはすでに非常に賢い。** 既に持っていないコンテキストのみ追加する。

### 適切な自由度を設定する
タスクの脆弱性・変動性に具体性を合わせる：
- **高い自由度**（テキスト指示）: 複数アプローチが有効、判断がコンテキスト依存（例: コードレビュー手順）
- **中程度**（疑似コード・パラメータ付きスクリプト）: 推奨パターンがあり多少のバリエーション可
- **低い自由度**（固定スクリプト）: 脆くエラーが起きやすい、一貫性・順序が重要（例: DBマイグレーション）

## フロントマター要件

| フィールド | 必須 | ルール |
|-----------|------|--------|
| `name` | ✅ | 最大64文字、小文字・数字・ハイフンのみ、XMLタグ不可、予約語（anthropic, claude）不可 |
| `description` | ✅ | 空不可、最大1024文字、XMLタグ不可、何をするか・いつ使うかを記述 |
| `allowed-tools` | ❌ | 許可なしで使えるツール（必要最小限） |
| `model` | ❌ | sonnet / opus / haiku / inherit |
| `context` | ❌ | `fork` でサブエージェント実行 |
| `user-invocable` | ❌ | スラッシュメニュー表示（デフォルト true） |
| `disable-model-invocation` | ❌ | モデル自動発動を無効化（デフォルト false） |

### 可視性制御

| 設定 | スラッシュメニュー | 自動検出 | 用途 |
|------|-----------------|---------|------|
| `user-invocable: true`（デフォルト） | 表示 | ○ | 明示呼び出し可能 |
| `user-invocable: false` | 非表示 | ○ | Claude自動判断のみ（前提知識の提供等） |
| `disable-model-invocation: true` | 表示 | ✗ | ユーザー明示呼び出しのみ（自動発動させたくない） |

## 命名規則

- 小文字・数字・ハイフンのみ（kebab-case）
- **動名詞形を推奨**（動詞 + -ing）: `processing-pdfs`, `analyzing-spreadsheets`, `testing-code`
- 許容代替: 名詞句（`pdf-processing`）、アクション指向（`process-pdfs`）
- **避ける**: 曖昧名（`helper`, `utils`, `tools`）、過度に汎用（`documents`, `data`）、予約語

> 補足: 既存のプライベートスキル群は動詞形（`create-pr`, `update-pr` 等）が多い。コレクション内の一貫性を優先してよい。

## 効果的な description（最重要）

Claudeは **description だけ** を読んで自動選択する。

### 必須3要素
1. **何をするか**: 具体的なアクション（抽出・入力・マージ・分析）
2. **いつ使うか**: 使用タイミング・トリガーコンテキスト
3. **トリガー用語**: 発動させたいキーワードを自然に含める

### 常に三人称で書く
- ✅ 「Excelファイルを処理してレポートを生成します」
- ❌ 「Excelファイルの処理をお手伝いできます」（一人称）
- ❌ 「これを使ってExcelファイルを処理できます」（二人称）

### 良い例
```yaml
description: PDFファイルからテキストや表を抽出し、フォーム入力やドキュメント結合を行う。PDFファイルの処理を依頼された時、または「PDF」「フォーム」「抽出」などのキーワードが含まれる場合に使用。
```

### 悪い例
```yaml
description: Helps with documents   # 曖昧すぎて選択されない
```

## progressive disclosure（段階的開示）

- **SKILL.md 本体は 500行未満**に保つ。近づいたら別ファイルへ分割
- SKILL.md は「目次」として機能し、必要時にClaudeが詳細ファイルを読む
- **参照は1レベルの深さに保つ**（ネストした参照は部分読みされ情報が欠落する）
- 100行超の参照ファイルは先頭に目次を付ける

### ディレクトリ構造例
```
my-skill/
├── SKILL.md          # 概要・クイックスタート（500行未満）
├── REFERENCE.md      # 詳細リファレンス
├── EXAMPLES.md       # 使用例
└── scripts/          # ユーティリティスクリプト（実行用、読み込み不要）
```

## ワークフローとフィードバックループ

### 複雑なタスクはチェックリスト化
コピーして進捗を追える形式を提供：
```
タスク進捗:
- [ ] ステップ1: フォームを分析
- [ ] ステップ2: マッピング作成
- [ ] ステップ3: 検証
```

### フィードバックループ
**バリデータ実行 → エラー修正 → 繰り返す**。出力品質を大きく向上させる。
重要操作（バッチ・破壊的変更）は「計画 → 検証 → 実行 → 確認」で中間出力を検証する。

## コンテンツガイドライン

- **時間依存情報を避ける**: 「2025年8月前は…」のような記述は避け、「古いパターン」セクション（`<details>`）に隔離
- **一貫した用語**: 1つの用語に統一（「field」と「box」「element」を混在させない）
- **具体的な例**: 抽象的な説明より入力/出力ペアを示す

---

## 避けるべきアンチパターン

- **Windowsスタイルのパス**: 常にスラッシュ（`scripts/helper.py`）。バックスラッシュ不可
- **選択肢の出しすぎ**: デフォルトを1つ示し、必要なら逃げ道を添える（「OCRが必要な場合のみ…」）
- **ツールが入っている前提**: 必要パッケージは明示しインストール手順を書く
- **Claudeへの丸投げ**（スクリプト）: エラー条件を明示的に処理する
- **マジックナンバー**: 設定値は理由をコメントで正当化する
- **MCPツールの非修飾名**: `ServerName:tool_name` の完全修飾名を使う
