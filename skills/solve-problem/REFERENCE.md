# 問題解決プロセス リファレンス

このファイルは問題解決プロセスの全体像と言葉の定義をまとめた参照資料です。

## 言葉の定義

### 基本概念

- **現状**: 現在の状態
- **あるべき姿**: 本来この状態であるべきという理想の姿
- **問題**: あるべき姿と現状のギャップ
- **問題解決**: 現状のレベルを上げて、あるべき姿とのギャップをなくすこと

### 思考法

- **ゼロベース思考**: 以下のような様々な前提や制約を自分の考えから外してものを見たり思考すること
  - **常識の枠**: 「それは当然のことだ。わかりきったことだ。」
  - **役割の枠**: 「それは自分の仕事ではない。それは他部署の仕事だ」
  - **経験の枠**: 「それは経験上関係ない（以前考えたが関係なかった）」
  - **意欲の枠**: 「その点には興味が湧かない。あまり関心がない。」

- **MECE**: 以下の例のように個々に見てダブりがなく、全体的に見て漏れがないこと
  - AとNot Aに切り分ける
    - 例：国内と海外。大手企業と大手企業以外。
  - 数式で分ける
    - 例：（売上）＝（社員数）×（社員1人あたりの売上高）
  - プロセスで分ける
    - 例：営業プロセス：訪問→商談→見積もり→成約
  - 一般的なフレームワークを用いる
    - 例：3C・4P・AIDMA
  - ※上記以外にもMECEの分け方はあり、目的に沿った意味ある切り口で分けることが大切

- **自責思考**: 自分たちで変えられるものに集中すること
  - 自責なもの：自分の行動、自分の意識
  - 他責なもの（注力すべきでない）：会社、他人、顧客

## 7フェーズ概要

### Phase 1: 問題の定義（What）

**目的**: 現状とあるべき姿のギャップを明確化

**主な活動**:
- 現状を把握する（事実ベース、状態で記載）
- あるべき姿を明確にする（定量的に）
- 問題（ギャップ）を明確にする

**成果物**: 現状・あるべき姿・問題の3要素

**詳細**: [PHASE-1-2.md](PHASE-1-2.md)

---

### Phase 2: 所在の特定（Where）

**目的**: 問題をMECEに分解し×○判定

**主な活動**:
- 問題をピラミッド構造に整理する（MECE分解）
- 問題の有無を判定する（×：問題あり、○：問題なし）

**成果物**: ピラミッド構造の問題分解図

**詳細**: [PHASE-1-2.md](PHASE-1-2.md)

---

### Phase 3: 原因追及・課題設定（Why）

**目的**: なぜを繰り返して真因と課題を抽出

**主な活動**:
- 「なぜ」を5回（複数回）繰り返す
- 真因の裏返しとなる課題を設定する
- 因果関係の双方向性を検証する

**成果物**: 原因追及ツリーと課題

**詳細**: [PHASE-3.md](PHASE-3.md)

---

### Phase 4: 解決策立案（How）

**目的**: 複数案の評価表で絞り込み

**主な活動**:
- 解決策を幅広く洗い出す
- 評価軸を設定する（効果、コスト、リスク、時間など）
- 3段階数値評価で絞り込む

**成果物**: 解決策評価表と選択理由

**詳細**: [PHASE-4.md](PHASE-4.md)

---

### Phase 5: タスク分解（Action）

**目的**: 解決策を1-3日粒度のタスクに分解

**主な活動**:
- 解決策を大きな塊で捉える
- 1-3日程度のタスクに分解する
- 漏れやすいタスクを意識する（修正、レビュー、周知、準備・後片付け）
- 工数を見積もる

**成果物**: タスク一覧と合計工数

**詳細**: [PHASE-5-6.md](PHASE-5-6.md)

---

### Phase 6: スケジュール作成（Schedule）

**目的**: マイルストーン設定と工数配置

**主な活動**:
- マイルストーンを設定する（3つの条件）
- タスクの順序・制約を洗い出す
- スケジュールを作成する（マイルストーンからの逆算）

**成果物**: ガントチャート形式のスケジュール

**詳細**: [PHASE-5-6.md](PHASE-5-6.md)

---

### Phase 7: リスク計画（Risk）

**目的**: リスク洗い出し、重みづけ、対策検討

**主な活動**:
- リスクを洗い出す（4要素：品質、範囲、時間、リソース）
- 発生確率×影響で重みづけする（2×2マトリクス）
- 予防策・発生時対策を検討する

**成果物**: リスク評価表と対策一覧

**詳細**: [PHASE-7.md](PHASE-7.md)

---

## プロセス全体の流れ

```
Phase 1: 問題の定義（What）
   ↓ 問題が明確化された
Phase 2: 所在の特定（Where）
   ↓ 問題の所在が×で特定された
Phase 3: 原因追及・課題設定（Why）
   ↓ 真因が特定され、課題が設定された
Phase 4: 解決策立案（How）
   ↓ 解決策が選択された
Phase 5: タスク分解（Action）
   ↓ タスクと工数が明確になった
Phase 6: スケジュール作成（Schedule）
   ↓ スケジュールが作成された
Phase 7: リスク計画（Risk）
   ↓ リスクと対策が明確になった
【実行】
```

## フレームワーク

### QCD（品質、コスト、納期）

原因追及や解決策検討で活用:
- **品質（Quality）**: 成果物の質に関する観点
- **コスト（Cost）**: 費用・リソースに関する観点
- **納期（Delivery）**: 時間に関する観点

### 4M（人、機械、方法、材料）

原因追及で網羅的に考える際に活用:
- **人（Man）**: 人的要因
- **機械（Machine）**: 設備・ツールの要因
- **方法（Method）**: プロセス・手順の要因
- **材料（Material）**: 素材・情報の要因

## よくある質問

**Q: 「なぜ」は必ず5回繰り返す必要がありますか？**
A: いいえ。5回は目安です。真因に到達すれば3回でも7回でも構いません。

**Q: 自責思考とは、すべて自分のせいにすることですか？**
A: いいえ。「自分たちで変えられるものに集中する」という意味です。他責なものに注力しても解決できないためです。

**Q: 評価軸の獲得ポイントは合計してはいけないのですか？**
A: はい。評価軸の優先順位に基づいて判断します。合計すると重要な評価軸が埋もれる可能性があります。

**Q: タスクの粒度が1-3日というのは厳密ですか？**
A: 目安です。管理可能なレベル（担当者が完了条件をイメージできるレベル）まで分解することが重要です。

**Q: すべてのフェーズを必ず実行する必要がありますか？**
A: はい。solve-problem-executorは全7フェーズを一括実行します。ドラフトファーストで全体像を作成してから調整する方針です。
