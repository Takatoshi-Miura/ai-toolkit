## Phase 5: タスク分解（Action）

### 概要

解決策を実行可能なタスクに分解する。

### 実行手順

#### 5-1. 解決策を大きな塊で捉える

**基本原則**:
- Phase 4で策定した解決策を実施するために必要な作業を俯瞰する
- まずは大きなカテゴリや工程で捉えてから、細分化していく

**大きな塊の例**:
- 準備フェーズ
- 実行フェーズ
- 検証フェーズ
- 展開フェーズ

#### 5-2. タスクを適切な粒度で分解する

**分解の粒度の基準**:
- 分解の粒度は管理や分担のしやすさで決める
- タスクの担当者が完了条件をイメージできるレベル（管理可能なレベル）まで分解する
- **1タスクは1〜3日程度で完了できるサイズを目安とする**

**粗すぎる場合の問題点**:
- 進捗がわかりづらくなる
- タスク完了の判断が難しい

**細かすぎる場合の問題点**:
- タスク同士の関係がわかりにくくなる
- 管理コストが増大する

#### 5-3. 漏れやすいタスクを意識する

**4つの漏れやすいタスク**:

1. **修正作業**
   - 一発OK前提だとリスクが高いため、修正作業をタスクに含めること
   - 例：「レビュー指摘事項の修正」「テスト不具合の修正」

2. **レビュー・承認**
   - 社内の業務プロセスに留意すること
   - 例：「上司レビュー」「関係部署承認」「品質チェック」

3. **周知**
   - 関係者への周知が必要ならタスクに含めること
   - 例：「関係者への説明会」「マニュアル配布」「キックオフミーティング」

4. **準備・後片付け**
   - 準備作業や後片付け作業も忘れずに含めること
   - 例：「環境準備」「データバックアップ」「旧システム撤去」

#### 5-4. 各タスクの工数を見積もる

**見積もりの方法**:
- 過去の類似作業を参考に見積もる
- 不確実性が高い場合は、バッファを設ける
- 楽観値・悲観値・最頻値の3点見積もりも有効

**工数の単位**:
- 時間単位（1時間、2時間など）
- 日単位（0.5日、1日、2日など）
- 人日単位（1人日、2人日など）

Phase 5を完了する前に、以下を確認してください:

- [ ] 大きな塊で捉えてから細分化できているか
- [ ] タスクの粒度は管理しやすいレベルか（1-3日程度）
- [ ] 修正作業がタスクに含まれているか
- [ ] レビューや承認プロセスがタスクに含まれているか
- [ ] 関係者への周知タスクが含まれているか
- [ ] 準備作業や後片付け作業が含まれているか
- [ ] 各タスクの工数が見積もられているか

## Phase 6: スケジュール作成（Schedule）

### 概要

タスクの順序・制約を整理し、実行可能なスケジュールを作成する。

### 実行手順

#### 6-1. タスクの順序・制約を洗い出す

**マイルストーンを設定する**:

マイルストーンとは：
- 進捗や必要条件を満たしているかを確認したり、軌道修正を図ったりする重要なポイント
- 遅延や品質低下のリスク・影響度を下げるために必要

**マイルストーンに適しているもの（3つの条件）**:
1. **時間軸（イベント・タイミング）に関わるもの**
   - 例：「年度末」「四半期末」「リリース日」
2. **承認をはじめ、後工程に影響の大きいもの**
   - 例：「基本設計承認」「本番環境構築完了」
3. **関係者が多く早めに調整が必要なもの**
   - 例：「キックオフミーティング」「全社説明会」

**タスクの流れを把握する**:
- 各タスクの前工程で何をしなければならないかを検討する
- タスク間の依存関係を明確にする（A完了後にB開始、など）
- 並行して実施可能なタスクを特定する（AとBは同時進行可能、など）

#### 6-2. スケジュールを作成する

**以下の3ステップで作成**:

1. **マイルストーンの時期を仮決めする**
   - 重要なイベントや制約条件を考慮して決定
   - 例：「基本設計承認：3/15」「リリース：4/1」

2. **各タスクの時期をマイルストーンからの逆算で仮決めしていく**
   - マイルストーンから逆算してタスクの開始日・終了日を決定
   - 依存関係を考慮してスケジュールを組む

3. **制約を変更できる可能性を考慮して1,2を繰り返す**
   - スケジュールが厳しい場合は、マイルストーンの時期を調整
   - タスクの並行実施やリソース追加を検討

#### 6-3. 考慮すべき制約

**3つの制約**:

1. **工数**: 各タスクに必要な作業時間
   - タスクの工数を積み上げて全体工数を算出
   - 工数が不足する場合はリソース追加を検討

2. **リソースバランス**: 担当者の稼働状況、同時並行作業の限界
   - 特定の担当者に負荷が集中していないか確認
   - 同時並行できるタスク数を考慮（通常は2-3個が限界）

3. **スケジュール**: 開催可能なタイミングが限定的なもの
   - 会議は特定の曜日・時間帯でしか開催できない
   - レビューは週1回しか実施できない
   - 関係者の休暇や他のプロジェクトの影響

## よくある質問

**Q: マイルストーンは何個設定すればよいですか？**
A: プロジェクトの規模によりますが、一般的には3-5個程度が適切です。多すぎると管理が煩雑になります。

**Q: すべてのタスクの依存関係を明記する必要がありますか？**
A: 主要な依存関係のみで構いません。すべてを明記すると複雑になりすぎます。

**Q: スケジュールが厳しい場合はどうすればよいですか？**
A: 以下の選択肢を検討してください：
  1. マイルストーンの時期を調整
  2. タスクの並行実施
  3. リソース追加（人員増強）
  4. スコープ調整（機能削減）

**Q: バッファはどの程度設けるべきですか？**
A: 不確実性の高いタスクには10-30%程度のバッファを設けることを推奨します。
