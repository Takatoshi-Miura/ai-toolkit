# Phase 7: リスク計画（Risk）

このファイルはPhase 7の詳細ガイドです。特定フェーズの修正が必要な場合に参照してください。

---

## 概要

リスクを洗い出し、予防策・発生時対策を検討する。

## 実行手順

### 7-1. リスクの洗い出し

**基本原則**:
- **マイルストーン遅れは全体の遅れに直結するため、マイルストーンに悪影響を及ぼす事象から優先的に検討する**
- **4要素（品質、範囲、時間、リソース）に悪影響を与え得ることを網羅的に検討する**

**リスク洗い出しの3つのヒント**:

1. **代替手段が乏しいタスクはないか**
   - そのタスクが失敗した場合、他の方法で挽回できるか？
   - コスト・時間投下での挽回が困難なタスクはリスクが高い
   - 例：特定の専門家にしかできないタスク、特殊な設備が必要なタスク

2. **KGI・KPIに悪影響を与えるタスクはないか**
   - そのタスクが失敗した場合、重要な指標に影響するか？
   - 例：売上に直結するタスク、顧客満足度に影響するタスク

3. **それが起こると全て台無しになるリスクはないか**
   - そのリスクが発生した場合、プロジェクト全体が失敗するか？
   - 例：予算が大幅に削減される、キーパーソンが退職する

**4要素での網羅的検討**:

1. **品質**: 成果物の質に関するリスク
   - 例：要求仕様を満たさない、不具合が多発する

2. **範囲**: プロジェクトのスコープに関するリスク
   - 例：スコープが拡大する、必要な機能が実現できない

3. **時間**: スケジュールに関するリスク
   - 例：マイルストーンに遅れる、納期に間に合わない

4. **リソース**: 人員・予算・設備に関するリスク
   - 例：担当者が不足する、予算が削減される、設備が故障する

### 7-2. リスクの重みづけ

**2×2マトリクスで評価**:

洗い出した各事象について、以下のスコアをつける:
- **発生確率**: どのくらいの頻度で起こりそうか（**大・小** の2段階評価）
- **影響**: 発生した場合の影響の大きさ（**大・小** の2段階評価）

**評価基準**:
- 発生確率「大」：50%以上の確率で発生しそう
- 発生確率「小」：50%未満の確率でしか発生しなさそう
- 影響「大」：プロジェクト全体に大きな影響を及ぼす
- 影響「小」：プロジェクトの一部にしか影響しない

**評価に応じた4パターンで分類**:

1. **発生確率：大、影響：大** → **最優先**
   - 予防策／発生時対策ともに検討する
   - 最も注意が必要なリスク

2. **発生確率：大、影響：小** → **優先度：中**
   - 1を検討した上で余裕があれば検討する
   - 発生しやすいが影響は限定的

3. **発生確率：小、影響：大** → **優先度：高**
   - 発生時対策を優先して検討する
   - 発生頻度は低いが、発生すると致命的

4. **発生確率：小、影響：小** → **優先度：低**
   - 対策検討の必要なし
   - リソースを割く価値がない

### 7-3. 予防策／発生時対策の検討

**3つの対策**:

1. **予防策**: リスク事象の発生確率を減らす対策のこと
   - 例：「担当者が不足するリスク」→ 予防策「事前に複数人で作業内容を共有しておく」
   - 例：「不具合が多発するリスク」→ 予防策「開発段階でテストを充実させる」

2. **発生時対策**: リスク事象が発生した場合の影響を軽減する対策のこと
   - 関係者と事前に練っておくことで有効性が高まる
   - 例：「担当者が不足するリスク」→ 発生時対策「他部署から応援を依頼する」
   - 例：「マイルストーンに遅れるリスク」→ 発生時対策「スコープを調整して一部機能を延期する」

3. **トリガーポイント**: 発生時対策を実行する条件のこと
   - いつ発生時対策を実行するかを明確にする
   - 例：「進捗が予定より3日遅れた時点」
   - 例：「不具合件数が10件を超えた時点」

**予定通りに進まない場合の対策例**:
- スコープ調整（機能削減）
- リソース追加（人員増強）
- スケジュール調整（納期延期）
- 品質基準の見直し

## よくある質問

**Q: すべてのリスクに予防策と発生時対策の両方を検討する必要がありますか？**
A: いいえ。2×2マトリクスの評価に応じて優先順位をつけてください。「発生確率：小、影響：小」のリスクは対策検討不要です。

**Q: リスクは何個洗い出せばよいですか？**
A: 最低でも4要素（品質、範囲、時間、リソース）ごとに1つずつ、合計4つ以上を推奨します。

**Q: トリガーポイントが明確に定義できない場合はどうすればよいですか？**
A: 「定期的に状況を確認し、必要に応じて発動する」という形でも構いません。ただし、できるだけ具体的な条件を設定することを推奨します。

**Q: 予防策を実施すれば、発生時対策は不要ですか？**
A: いいえ。予防策を実施しても100%リスクを防げるわけではありません。「発生確率：大、影響：大」のリスクには両方の対策が必要です。

## 実例

### 良い例

```markdown
## Phase 7: リスク計画（Risk）

### リスク評価と対策

| 分類 | 事象 | 発生確率 | 影響 | 予防策 | 発生時対策 | トリガーポイント |
|---|---|---|---|---|---|---|
| 時間 | 説明会の参加者が集まらない | 大 | 大 | 事前に複数回のリマインドを送信 | 個別説明会を追加開催 | 参加申込が定員の50%を下回った時点 |
| リソース | 担当者が体調不良で休む | 小 | 大 | - | 他のメンバーが代行できるよう事前に作業内容を共有 | 担当者が2日以上休んだ時点 |
| 品質 | ヒアリングシートの活用率が低い | 大 | 小 | 活用方法の説明会を実施 | 個別フォローアップを実施 | 活用率が50%を下回った時点 |
| 範囲 | 上司から追加要望が出る | 小 | 小 | - | - | - |

### リスク対応の優先順位
1. 【最優先】時間リスク（発生確率：大、影響：大）
2. 【優先度：高】リソースリスク（発生確率：小、影響：大）
3. 【優先度：中】品質リスク（発生確率：大、影響：小）
4. 【優先度：低】範囲リスク（発生確率：小、影響：小）

- 上記に設定した理由：マイルストーン「説明会」への影響を優先的に考慮し、4要素で網羅的にリスクを洗い出した
```
