# daily-notebooklm-research

NotebookLMの音声解説（Audio Overview）に渡すURLリストとプロンプトを、毎朝自動で生成する。

日本語で回答すること。

## 重要：このスキルの使い方

**各フェーズを順番に実行すること。フェーズを飛ばしてはならない。**

**制約事項：**
- 調査対象の決定は2段階: まずCONFIG.mdの未調査テーマリストを確認し、なければGoogleドキュメント・過去セッションログ・ai-toolkit既存リソースの3方向を**すべて見て**総合判断する（先に見つかったものに飛びつかない）
- 他スキル（`google-drive-skill`等）は参照・呼び出ししない。Googleドキュメント読み取りは本スキル内の `scripts/read_doc.py` を直接実行する（依存を持たない自己完結構成）
- セッションログは `~/Documents/ClaudeLogs/all-projects/` 配下の既存mdを読むのみ。新規生成はしない（トークン消費を避けるため）。存在しなければスキップする
- 出力前のセルフレビューは最大3回までとし、3回経っても解決しない項目は「未解決事項」として正直に報告する

## Phase 1: CONFIG.md読み込み・未調査テーマの確認

`~/.claude/skills/daily-notebooklm-research/CONFIG.md` を読み込み、「未調査テーマリスト」セクションを確認する。

- **1件以上ある場合**: 先頭のテーマを調査対象として確定 → Phase 3へ
- **0件の場合**: Phase 2のフォールバック調査へ

**成功確認**: 調査対象が確定した、またはフォールバックが必要と判断した → 次のPhaseへ

---

## Phase 2: 調査対象の決定（フォールバック）

候補収集（2-1・2-2・2-3）をすべて行い、出てきた候補をまとめて比較した上で、最も優先度が高いテーマを1つ選ぶ（直列の先勝ちフォールバックではない）。

### 2-1. Googleドキュメントの確認

CONFIG.mdの「Googleドキュメント参照先」に記載のURLから `{fileId}`（`/d/{fileId}/edit` 部分）を抽出し、以下を実行する：

```bash
python3 ~/.claude/skills/daily-notebooklm-research/scripts/read_doc.py <fileId>
```

引数を省略すると、当月タブ（`YYYYMM`形式、例: `202606`）を自動選択して読み取る。タブが存在しないドキュメントの場合は本文全体を読む。当月タブが見つからない場合はドキュメント内の最初のタブにフォールバックする。

タブ構成を確認したい場合は `python3 ~/.claude/skills/daily-notebooklm-research/scripts/read_doc.py <fileId> --list-tabs` でタブ一覧（タイトル・ID）を取得できる。特定のタブを指定したい場合は `python3 ~/.claude/skills/daily-notebooklm-research/scripts/read_doc.py <fileId> <tabTitle>` を使う。

`success: true` の場合は `text` フィールドの内容から候補テーマを洗い出す。`success: false` の場合は本ファイル末尾の「エラー対応」表を参照しつつ、このソースは候補なしとして次に進む。

### 2-2. 過去セッションログの確認

`~/Documents/ClaudeLogs/all-projects/` 配下に既存のmdファイルがあれば、最も新しいものを1つ読み、候補テーマを洗い出す。存在しなければ候補なしとして次に進む（このスキルから新規生成は行わない）。

### 2-3. ai-toolkit既存リソースの確認

`~/Documents/Git/ai-toolkit` の既存リソース（スキル・エージェント・スクリプト）を眺め、改善余地がありそうな箇所があれば、それに関連する調査テーマを候補に加える。

### 2-4. 総合判断による選定

2-1・2-2・2-3で洗い出した候補をすべて並べ、`history.md` の調査済みテーマと重複するものを除外する。残った候補の中から、CONFIG.mdの「テーマ選定時の心がけ」に従って最も優先度が高い1つを選ぶ。

**候補が1件もない場合**: 「調査対象を決定できませんでした」とユーザーに報告し、終了する。

**成功確認**: 調査対象テーマが1つ確定した → Phase 3へ

## Phase 3: Web調査

CONFIG.mdの「ソース選定時の心がけ」を読み込み、その項目に従ってWebSearch/WebFetchで調査する。

**成功確認**: 心がけ項目を満たすURLと説明が揃った → Phase 4へ

---

## Phase 4: 出力生成

### 4-1. URLリストの生成

NotebookLMにそのまま貼り付けられるよう、コードブロックには **URLのみ**（番号・説明文を含めない）を1行ずつ記載する。各URLの一言説明は、コードブロックの外側に「URLリストと対応する説明」として別途記載する（説明文がコピペ時に混入しないようにするため）。

### 4-2. NotebookLM用プロンプトの生成

CONFIG.mdの「音声解説プロンプト作成時の心がけ」を読み込み、その項目に従ってコピペしやすいコードブロック形式でプロンプトを生成する。

**成功確認**: URLリストとプロンプトの草案ができた → Phase 4.5へ

---

## Phase 4.5: セルフレビュー＆フィードバックループ（最大3回）

CONFIG.mdの「ソース選定時の心がけ」「音声解説プロンプト作成時の心がけ」の各項目をそのままチェックリスト化し、生成物が満たしているかを1項目ずつ検証する。

### 4.5-1. URL有効性チェック

「各URLは実際にアクセス可能であること」の項目は、以下のいずれかでURLごとに機械的に検証する（推測で判定しない）：

```bash
curl -o /dev/null -s -w "%{http_code}\n" -L --max-time 10 "<URL>"
```

- ステータスコードが200〜399 → 有効
- 400番台・500番台、タイムアウト、名前解決エラー → 無効。当該URLは候補から外し、Phase 3に戻って代替ソースを探す
- curlが使えない/ブロックされる環境では、同等の判定としてWebFetchで取得し、エラーが返らないことを確認する

### 4.5-2. その他のチェックリスト検証

URL有効性チェック以外の心がけ項目（件数・視点の多様性・説明の有無・プロンプトの要件など）を1項目ずつ検証する。

**不備がある場合**: 該当箇所を修正し、再度チェックリストを検証する（試行回数+1）。

**終了条件**:
- 全項目を満たした → Phase 5へ（このとき試行回数も記録する）
- 3回修正しても解決しない項目がある → 「未解決事項」として明記し、Phase 5へ進む

**成功確認**: 全項目OK、または3回到達した → Phase 5へ

---

## Phase 5: チャット表示・ファイル保存

### 5-1. チャット表示

URLのみのコピペ用コードブロック、各URLの説明（コードブロック外）、NotebookLM用プロンプトのコピペ用コードブロックを、それぞれ表示する。未解決事項があれば併記する。

### 5-2. ファイル保存

`~/.claude/skills/daily-notebooklm-research/output/<YYYY-MM-DD>_<テーマ>.md` に同内容を保存する。

**成功確認**: チャット表示とファイル保存が完了した → Phase 6へ

---

## Phase 6: 履歴更新

### 6-1. history.mdへの追記

`日付・テーマ・出力ファイルパス・セルフレビュー試行回数` を1行追記する。

### 6-2. CONFIG.mdの未調査テーマリストの更新

Phase 1で使用したテーマが「未調査テーマリスト」由来だった場合、そのテーマをリストから削除する。

**成功確認**: history.mdとCONFIG.mdの更新が完了した → 完了

## エラー対応

| エラー | 対応 |
|-------|------|
| CONFIG.mdが見つからない | スキルディレクトリにCONFIG.mdが存在するか確認する |
| Googleドキュメントが読めない（認証エラー等） | [SETUP.md](SETUP.md) の認証セットアップワークフローを実行する |
| セッションログmdが存在しない | このソースは候補なしとしてPhase 2-3に進む（このスキルからの新規生成は行わない） |
| 調査対象が決定できない | ユーザーに報告し終了する。CONFIG.mdへのテーマ追記を提案する |
| セルフレビューが3回経っても解決しない | 未解決事項として明記し、そのまま出力する（無限に繰り返さない） |
| URLがアクセス不可（404・タイムアウト等） | 候補から除外し、Phase 3に戻って代替ソースを探す。3回試行しても代替が見つからない場合は未解決事項として明記する |

**エラーフィードバックループ**:
1. エラー内容を確認
2. 上記表に従って対応
3. 該当フェーズを再実行
4. 成功するまで繰り返す（セルフレビューは最大3回まで）

## 出力形式

- チャット上にURLのみのコピペ用コードブロック（番号・説明なし）、各URLの説明（コードブロック外）、NotebookLM用プロンプトのコピペ用コードブロックを表示する
- 同内容を `output/<YYYY-MM-DD>_<テーマ>.md` に保存する

## 注意事項

- CONFIG.mdは人が日常的に編集する想定。未調査テーマの追加・心がけ項目の調整はCONFIG.mdのみで完結する
- history.mdは自動追記のみ。手動編集は調査済みテーマを誤って消さない範囲で行う
