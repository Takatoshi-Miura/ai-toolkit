## Phase 1: 要件ヒアリング

### 1-1. ユーザーへの質問

AskUserQuestionツールで以下を収集:

| 項目 | 必須 | 例 |
|------|------|-----|
| 対応したいこと | ✅ | 新機能追加、バグ修正、UI改善など |
| 対象画面・機能 | ❌ | 課題一覧、ノート追加など |
| 参考にすべき仕様書 | ❌ | doc/01_task_list.md など |
| 優先度・制約 | ❌ | 既存データとの互換性、パフォーマンス要件など |

### 1-2. 仕様書の確認

対象画面・機能が特定された場合、関連する仕様書を読み取る:

1. **アプリ全体の仕様概要**: `~/Documents/Git/SportsNote_iOS/doc/00_overview.md` を必ず確認
2. **該当画面の仕様書**: `~/Documents/Git/SportsNote_iOS/doc/` から関連ファイルを読み取り

### 1-3. 情報の整理

収集した情報を整理し、ユーザーに確認:
- 対応内容の要約
- 対象画面・機能の特定
- 関連する仕様書の内容要点

**成功確認**: 必要な情報がすべて揃い、ユーザーが要件を確認 → Phase 2へ

---

## Phase 2: 調査・実装計画

### 2-1. Plan サブエージェントの呼び出し

Taskツールで `Plan` サブエージェントを起動:

```
subagent_type: "Plan"
prompt: |
  以下の情報に基づいて実装計画を立ててください。

  ## 実装内容
  {Phase 1でヒアリングした要件}

  ## 参照情報
  - CLAUDE.md: ~/Documents/Git/SportsNote_iOS/CLAUDE.md
  - 仕様書: {関連する仕様書の内容}

  ## 作成してほしいもの
  1. 不明点があればユーザーに質問（解消するまで繰り返し）
  2. 既存の類似実装パターン調査（~/Documents/Git/SportsNote_iOS/ 内）
  3. 修正対象ファイルの特定
  4. 実装方針の策定
```

### 2-2. 実装計画の出力フォーマット

Planサブエージェントは以下の形式で出力:

```markdown
## 実装計画

### 1. 概要
- **目的**: （1-2文）
- **スコープ**: （対象範囲）
- **除外事項**: （今回対象外）

### 2. 関連する仕様書
- （参照した仕様書名とパス）

### 3. 前提条件
- （MVVM構成、Realm/Firebase利用等）

### 4. 修正対象ファイル
| ファイルパス | 変更種別 | 変更概要 |
|-------------|---------|---------|

### 5. 実装ステップ
#### Step N: （ステップ名）
- **対象ファイル**: path/to/file.swift
- **変更内容**: （箇条書き）
- **依存関係**: （他ステップとの依存）

### 6. 類似実装の参照
### 7. リスク・注意点
### 8. 仕様書との整合性確認
```

### 2-3. ユーザー承認

1. 実装計画をユーザーに提示
2. スコープ・修正対象・実装ステップ・リスクの観点で承認を求める
3. 承認されない場合 → 2-1に戻り計画を修正
4. 承認された場合 → Phase 3へ

**成功確認**: ユーザーが実装計画を承認 → Phase 3へ

---

## Phase 3: 実装・ビルド・テスト

### 3-1. コード実装

承認された実装計画に従い、直接コードを実装する:

1. CLAUDE.mdのコーディングルールに従う
2. Edit/Writeツールでコードを変更・作成
3. 既存コードのスタイルを尊重する

### 3-2. swift-format の実行

CLAUDE.mdに記載のコマンドでswift-formatを実行:
- 変更したファイルに対してswift-formatを適用
- フォーマットエラーがあれば修正

### 3-3. ビルド実行

CLAUDE.mdに記載のコマンドでビルドを実行:
- ビルドエラーがあれば修正し、再ビルド
- BUILD SUCCEEDEDになるまで繰り返す

### 3-4. テスト（条件付き）

**テスト不要**: 既存共通クラスの呼び出しのみ / 既存テストで十分カバー / UIの軽微な変更のみ

**テスト必要**: 新規ビジネスロジック / 複雑な条件分岐 / データモデルの変更

テストが必要な場合:
- テストディレクトリ: `~/Documents/Git/SportsNote_iOS/SportsNote_iOSTests/`
- テストフレームワーク: XCTest
- CLAUDE.mdに記載のテストコマンドで実行

**完了条件（すべて満たすこと）**:
- [ ] swift-format が正常に完了
- [ ] ビルドが成功（BUILD SUCCEEDED）
- [ ] テスト実行が必要な場合、テストが成功

**成功確認**: ビルド成功・テスト成功（必要な場合） → Phase 4へ

## Phase 5: ドキュメント更新・完了

### 5-1. ドキュメント更新の判定

以下に該当する場合、仕様書の更新を実施:

| 更新条件 | 対象ドキュメント |
|---------|----------------|
| 画面の表示内容・動作が変わった場合 | 該当画面の仕様書（`doc/XX_*.md`） |
| 新しい画面を追加した場合 | 新規仕様書の作成 + `00_overview.md` の画面一覧更新 |
| データモデルが変わった場合 | `00_overview.md` のデータモデルセクション |
| 画面遷移が変わった場合 | `00_overview.md` の画面遷移図 |

**更新不要の場合**: 軽微なバグ修正、リファクタリング、パフォーマンス改善など仕様に影響しない変更

### 5-2. ドキュメント更新の実施

- ドキュメントパス: `~/Documents/Git/SportsNote_iOS/doc/`
- 既存の仕様書のフォーマットに合わせて更新
- 新規仕様書は既存のナンバリング規則に従う（`NN_画面名.md`）

### 5-3. 総評の作成

以下の観点で報告:

```markdown
## 総評

### 今回のタスクでうまくいった点
- （箇条書き）

### 改善が必要だった点
- （箇条書き）

### 仕様書の更新
- （更新したドキュメント一覧、または「更新不要」）

### プロンプトへの修正提案（あれば）
- （具体的な修正提案）
```

## エラーハンドリング

| エラー | 対応 |
|-------|------|
| 情報不足 | AskUserQuestionで追加情報を収集 |
| 仕様書が見つからない | `doc/` のファイル一覧を表示し確認 |
| サブエージェント失敗 | エラーメッセージを分析し再試行（最大3回） |
| swift-format失敗 | CLAUDE.mdのコマンドを確認、エラーメッセージ表示 |
| ビルド失敗 | エラーメッセージを分析し修正支援 |
| テスト失敗 | 失敗テストを特定し修正 |
| 仕様書との不整合 | ユーザーに確認し、仕様書 or 実装を修正 |

## 注意事項

- 過剰な改善提案は避け、要求されたスコープに集中する
- 既存コードのスタイルを尊重する（CLAUDE.mdの規約に従う）
- ビルド・テストコマンドはCLAUDE.mdに記載のものをそのまま使用する
- 仕様書の更新はPhase 5で実施（Phase 3の実装中には行わない）
- 作成・変更前に必ずユーザーの承認を得る
